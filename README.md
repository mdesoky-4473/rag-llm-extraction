# RAG LLM Extraction

A Retrieval-Augmented Generation (RAG) service that lets users upload documents, index them, and ask questions grounded in the document content. Answers are generated by Claude with confidence scoring, source attribution, and configurable guardrails.

## Architecture

```
Frontend (HTML)                    Backend (FastAPI + Python)
┌─────────────────┐               ┌──────────────────────────────┐
│  Upload docs    │──POST /upload─▶│  Save to uploads/            │
│  Click Reindex  │──POST /reindex▶│  Chunk → TF-IDF → Vectors   │
│  Ask question   │──POST /ask────▶│  Retrieve → Prompt → Claude  │
│  View answer    │◀──── JSON ────│  Return answer + sources      │
└─────────────────┘               └──────────────────────────────┘
```

## How It Works

1. **Upload** — Users upload PDF, TXT, or DOCX files through the frontend
2. **Chunk** — Documents are split into ~300 character overlapping chunks
3. **Embed** — Chunks are vectorized using TF-IDF (500 dimensions, bigrams)
4. **Retrieve** — User query is embedded and compared via cosine similarity
5. **Generate** — Top matching chunks are sent to Claude as grounded context
6. **Guardrails** — Confidence thresholds, strict/flexible modes, source citations

## Features

- **Document upload** — Drag-and-drop or click to upload (PDF, TXT, DOCX)
- **Per-request API key** — Users provide their own Anthropic API key
- **Strict mode toggle** — Strict (no interpretation) vs flexible (allows inference)
- **Confidence scoring** — HIGH / MEDIUM / LOW based on retrieval + LLM response
- **Source attribution** — Every answer cites which documents were used
- **Reindexing** — Rebuild the search index when documents change
- **Auto-index** — Server indexes existing uploads on startup

## Setup

### Requirements

```
pip install fastapi uvicorn scikit-learn anthropic pdfplumber python-multipart
```

### Run

```bash
cd rag-be
python rag_service.py
```

Server starts on `http://localhost:8000`. Open `rag-fe/index.html` in a browser.

Interactive API docs available at `http://localhost:8000/docs`.

## Technical Details

- **Embedding**: TF-IDF with 500 max features, English stop words, unigrams + bigrams
- **Retrieval**: Cosine similarity, top-k (default 3)
- **Chunking**: 300 characters with 50 character overlap, sentence boundary detection
- **LLM**: Claude Sonnet via Anthropic API, temperature 0.1
- **Confidence threshold**: Score < 0.05 returns safe fallback without calling LLM
- **Validation**: Pydantic models enforce request/response schemas

## Production Considerations

In a production environment (e.g., healthcare AI at scale), this architecture would be extended with:

- **Neural embeddings** (Bedrock Titan, sentence-transformers) instead of TF-IDF for semantic understanding
- **Vector database** (pgvector, Pinecone, OpenSearch) instead of in-memory storage
- **Metadata filtering** — filter by organization, department, or policy version before retrieval
- **Service accounts** for document source authentication instead of OAuth
- **Event-driven ingestion** — auto-reindex when source documents change
- **Server-side API key** management via environment variables or AWS Secrets Manager
- **Evaluation pipeline** — precision/recall on golden datasets, drift detection, latency monitoring
